# Setup — Project Knowledge Initialization (Experimental)

Scan the project and create `.aria/` with structured project knowledge for other aria skills to consume.

**Announce at start:** "Using aria:setup to initialize project knowledge. (This feature is experimental.)"

## This is the ONLY skill that creates `.aria/`

No other aria skill is allowed to create the `.aria/` directory or its files. If `.aria/` doesn't exist and another skill is running, it must skip silently — never create it.

## Process

### Step 1: Pre-flight Check

```bash
ls -la .aria/ 2>/dev/null
```

- **If `.aria/` exists:** "`.aria/` already exists. Re-run setup to regenerate `project.md`? (`learnings.md` will be preserved.)"
  - Wait for confirmation before proceeding
- **If `.aria/` does not exist:** proceed

### Step 2: Automated Scan

Gather project information without asking the user:

**Tech stack detection:**
- Languages: check file extensions, count distribution
- Package managers: `package.json`, `*.csproj`, `requirements.txt`, `go.mod`, `Cargo.toml`, `pom.xml`, etc.
- Frameworks: scan config files and imports for known frameworks
- Test frameworks: scan test directories for test runner config or imports

**Project structure:**
- Identify source directories vs test directories vs config vs docs
- Find entry points (main files, startup files, API controllers)
- Detect monorepo vs single-project structure

**Existing documentation:**
- Read `CLAUDE.md` if it exists — note what's already documented (don't duplicate)
- Read `README.md` if it exists
- Check for `docs/`, `architecture/`, `.claude/` directories

**Build & test commands:**
- Extract from `CLAUDE.md`, `Makefile`, `package.json` scripts, CI config, etc.
- If found in `CLAUDE.md`, reference it ("see CLAUDE.md") instead of duplicating

### Step 3: Interactive Questions

Ask **one at a time**, skip if the answer is already obvious from the scan:

1. "What's the main purpose of this project?" *(skip if README/CLAUDE.md already answers)*
2. "Any areas of the codebase that are particularly fragile or complex?" *(always ask — not detectable by scan)*
3. "Any conventions or patterns not captured in CLAUDE.md or docs?" *(skip if no CLAUDE.md — everything is new)*
4. "Anything else aria should know about this project?" *(open-ended, always ask)*

### Step 4: Generate `.aria/project.md`

```markdown
# Project Knowledge — <project name>

*Generated by aria:setup on <date>. Re-run `aria:setup` to regenerate.*

## Overview

[One sentence purpose — from user or README]

## Structure

[Key directories and their roles — from scan]
[Only list what's NOT already in CLAUDE.md]

## Tech Stack

[Detected languages, frameworks, tools — concise list]

## Conventions

[Patterns detected + user-provided conventions]
[Reference CLAUDE.md for items already documented there: "See CLAUDE.md for [topic]"]

## Known Complexity

[Areas flagged by user as fragile/complex — this is the unique value of setup]
```

**Keep it concise.** This file is loaded into context by other skills, so every line costs tokens. Reference CLAUDE.md instead of duplicating. Target: under 50 lines.

### Step 5: Create `.aria/learnings.md`

Only if it doesn't already exist (preserve on re-run):

```markdown
# Learnings

*Captured by aria:learn. Each entry records a non-obvious insight from a work session.*
```

### Step 6: Show and Confirm

Display the generated `project.md` content to the user:

```
## .aria/ initialized

**project.md** (N lines):
[content]

**learnings.md:** empty template (populated by aria:learn)

Does this look right? Edit anything before I save?
```

Wait for the user to approve or request changes. Save only after confirmation.

### Step 7: Commit

```bash
git add .aria/
git commit -m "chore: initialize aria project knowledge"
```

## Re-run Behavior

When `.aria/` already exists:
- **Regenerate** `project.md` from scratch (re-scan + re-ask)
- **Preserve** `learnings.md` — never overwrite or delete it
- Show diff between old and new `project.md` before saving

## Key Rules

- **Concise output** — `project.md` should be under 50 lines. It's loaded every session.
- **Don't duplicate CLAUDE.md** — reference it. The value of `.aria/` is what CLAUDE.md doesn't cover.
- **Always ask about complexity** — that's the one thing no scan can detect.
- **Commit the result** — `.aria/` is meant to be shared via git.

## Integration

- **aria:learn** — writes to `.aria/learnings.md` created here
- **aria:design, aria:plan, aria:exec** — conditionally read `.aria/` at start (after Task 4 patches them)
